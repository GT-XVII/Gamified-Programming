function \+\_\+expand\+Key (\begin{DoxyParamCaption}\item[{}]{key}{, }\item[{}]{decrypt}{}\end{DoxyParamCaption})}

Generates a key schedule using the AES key expansion algorithm.

The AES algorithm takes the Cipher Key, K, and performs a Key Expansion routine to generate a key schedule. The Key Expansion generates a total of Nb\texorpdfstring{$\ast$}{*}(Nr + 1) words\+: the algorithm requires an initial set of Nb words, and each of the Nr rounds requires Nb words of key data. The resulting key schedule consists of a linear array of 4-\/byte words, denoted \mbox{[}wi \mbox{]}, with i in the range 0 \texorpdfstring{$<$}{<}= i \texorpdfstring{$<$}{<} Nb(Nr + 1).

Key\+Expansion(byte key\mbox{[}4\texorpdfstring{$\ast$}{*}\+Nk\mbox{]}, word w\mbox{[}Nb\texorpdfstring{$\ast$}{*}(Nr+1)\mbox{]}, Nk) AES-\/128 (Nb=4, Nk=4, Nr=10) AES-\/192 (Nb=4, Nk=6, Nr=12) AES-\/256 (Nb=4, Nk=8, Nr=14) Note\+: Nr=Nk+6.

Nb is the number of columns (32-\/bit words) comprising the State (or number of bytes in a block). For AES, Nb=4.

{\em decrypt} & true to modify the key schedule to decrypt, false not to.\\
\hline
\end{DoxyParams}

One way to understand how AES works is to imagine that \textquotesingle{}addition\textquotesingle{} and \textquotesingle{}multiplication\textquotesingle{} are interfaces that require certain mathematical properties to hold true (ie\+: they are associative) but they might have different implementations and produce different kinds of results ... provided that their mathematical properties remain true. AES defines its own methods of addition and multiplication but keeps some important properties the same, ie\+: associativity and distributivity. The explanation below tries to shed some light on how AES defines addition and multiplication of bytes and 32-\/bit words in order to perform its encryption and decryption algorithms.

The basics\+:

The AES algorithm views bytes as binary representations of polynomials that have either 1 or 0 as the coefficients. It defines the addition or subtraction of two bytes as the XOR operation. It also defines the multiplication of two bytes as a finite field referred to as GF(2\texorpdfstring{$^\wedge$}{\string^}8) (Note\+: \textquotesingle{}GF\textquotesingle{} means "{}\+Galois Field"{} which is a field that contains a finite number of elements so GF(2\texorpdfstring{$^\wedge$}{\string^}8) has 256 elements).

This means that any two bytes can be represented as binary polynomials; when they multiplied together and modularly reduced by an irreducible polynomial of the 8th degree, the results are the field GF(2\texorpdfstring{$^\wedge$}{\string^}8). The specific irreducible polynomial that AES uses in hexadecimal is 0x11b. This multiplication is associative with 0x01 as the identity\+:

(b \texorpdfstring{$\ast$}{*} 0x01 = GF(b, 0x01) = b).

The operation GF(b, 0x02) can be performed at the byte level by left shifting b once and then XOR\textquotesingle{}ing it (to perform the modular reduction) with 0x11b if b is \texorpdfstring{$>$}{>}= 128. Repeated application of the multiplication of 0x02 can be used to implement the multiplication of any two bytes.

For instance, multiplying 0x57 and 0x13, denoted as GF(0x57, 0x13), can be performed by factoring 0x13 into 0x01, 0x02, and 0x10. Then these factors can each be multiplied by 0x57 and then added together. To do the multiplication, values for 0x57 multiplied by each of these 3 factors can be precomputed and stored in a table. To add them, the values from the table are XOR\textquotesingle{}d together.

AES also defines addition and multiplication of words, that is 4-\/byte numbers represented as polynomials of 3 degrees where the coefficients are the values of the bytes.

The word \mbox{[}a0, a1, a2, a3\mbox{]} is a polynomial a3x\texorpdfstring{$^\wedge$}{\string^}3 + a2x\texorpdfstring{$^\wedge$}{\string^}2 + a1x + a0.

Addition is performed by XOR\textquotesingle{}ing like powers of x. Multiplication is performed in two steps, the first is an algebriac expansion as you would do normally (where addition is XOR). But the result is a polynomial larger than 3 degrees and thus it cannot fit in a word. So next the result is modularly reduced by an AES-\/specific polynomial of degree 4 which will always produce a polynomial of less than 4 degrees such that it will fit in a word. In AES, this polynomial is x\texorpdfstring{$^\wedge$}{\string^}4 + 1.

The modular product of two polynomials \textquotesingle{}a\textquotesingle{} and \textquotesingle{}b\textquotesingle{} is thus\+:

\doxylink{uuid_8min_8js_aeb337d295abaddb5ec3cb34cc2e2bbc9}{d(x)} = d3x\texorpdfstring{$^\wedge$}{\string^}3 + d2x\texorpdfstring{$^\wedge$}{\string^}2 + d1x + d0 with d0 = GF(a0, b0) \texorpdfstring{$^\wedge$}{\string^} GF(a3, b1) \texorpdfstring{$^\wedge$}{\string^} GF(a2, b2) \texorpdfstring{$^\wedge$}{\string^} GF(a1, b3) d1 = GF(a1, b0) \texorpdfstring{$^\wedge$}{\string^} GF(a0, b1) \texorpdfstring{$^\wedge$}{\string^} GF(a3, b2) \texorpdfstring{$^\wedge$}{\string^} GF(a2, b3) d2 = GF(a2, b0) \texorpdfstring{$^\wedge$}{\string^} GF(a1, b1) \texorpdfstring{$^\wedge$}{\string^} GF(a0, b2) \texorpdfstring{$^\wedge$}{\string^} GF(a3, b3) d3 = GF(a3, b0) \texorpdfstring{$^\wedge$}{\string^} GF(a2, b1) \texorpdfstring{$^\wedge$}{\string^} GF(a1, b2) \texorpdfstring{$^\wedge$}{\string^} GF(a0, b3)

As a matrix\+:

\mbox{[}d0\mbox{]} = \mbox{[}a0 a3 a2 a1\mbox{]}\mbox{[}b0\mbox{]} \mbox{[}d1\mbox{]} \mbox{[}a1 a0 a3 a2\mbox{]}\mbox{[}b1\mbox{]} \mbox{[}d2\mbox{]} \mbox{[}a2 a1 a0 a3\mbox{]}\mbox{[}b2\mbox{]} \mbox{[}d3\mbox{]} \mbox{[}a3 a2 a1 a0\mbox{]}\mbox{[}b3\mbox{]}

Special polynomials defined by AES (0x02 == \{02\})\+: \doxylink{ordered-map_8min_8js_aa4d4888597588a84fd5b1184d00c91f3}{a(x)} = \{03\}x\texorpdfstring{$^\wedge$}{\string^}3 + \{01\}x\texorpdfstring{$^\wedge$}{\string^}2 + \{01\}x + \{02\} a\texorpdfstring{$^\wedge$}{\string^}-\/1(x) = \{0b\}x\texorpdfstring{$^\wedge$}{\string^}3 + \{0d\}x\texorpdfstring{$^\wedge$}{\string^}2 + \{09\}x + \{0e\}.

These polynomials are used in the Mix\+Columns() and Inverse\+Mix\+Columns() operations, respectively, to cause each element in the state to affect the output (referred to as diffusing).

Rot\+Word() uses\+: a0 = a1 = a2 = \{00\} and a3 = \{01\}, which is the polynomial x3.

The Shift\+Rows() method modifies the last 3 rows in the state (where the state is 4 words with 4 bytes per word) by shifting bytes cyclically. The 1st byte in the second row is moved to the end of the row. The 1st and 2nd bytes in the third row are moved to the end of the row. The 1st, 2nd, and 3rd bytes are moved in the fourth row.

More details on how AES arithmetic works\+:

In the polynomial representation of binary numbers, XOR performs addition and subtraction and multiplication in GF(2\texorpdfstring{$^\wedge$}{\string^}8) denoted as GF(a, b) corresponds with the multiplication of polynomials modulo an irreducible polynomial of degree 8. In other words, for AES, GF(a, b) will multiply polynomial \textquotesingle{}a\textquotesingle{} with polynomial \textquotesingle{}b\textquotesingle{} and then do a modular reduction by an AES-\/specific irreducible polynomial of degree 8.

\doxylink{class_a}{A} polynomial is irreducible if its only divisors are one and itself. For the AES algorithm, this irreducible polynomial is\+:

\doxylink{agent-base_2node__modules_2ms_2index_8js_a9e77e016b2928d7dcb493b89a0c9dc32}{m(x)} = x\texorpdfstring{$^\wedge$}{\string^}8 + x\texorpdfstring{$^\wedge$}{\string^}4 + x\texorpdfstring{$^\wedge$}{\string^}3 + x + 1,

or \{01\}\{1b\} in hexadecimal notation, where each coefficient is a bit\+: 100011011 = 283 = 0x11b.

For example, GF(0x57, 0x83) = 0xc1 because

0x57 = 87 = 01010111 = x\texorpdfstring{$^\wedge$}{\string^}6 + x\texorpdfstring{$^\wedge$}{\string^}4 + x\texorpdfstring{$^\wedge$}{\string^}2 + x + 1 0x85 = 131 = 10000101 = x\texorpdfstring{$^\wedge$}{\string^}7 + x + 1

(x\texorpdfstring{$^\wedge$}{\string^}6 + x\texorpdfstring{$^\wedge$}{\string^}4 + x\texorpdfstring{$^\wedge$}{\string^}2 + x + 1) \texorpdfstring{$\ast$}{*} (x\texorpdfstring{$^\wedge$}{\string^}7 + x + 1) = x\texorpdfstring{$^\wedge$}{\string^}13 + x\texorpdfstring{$^\wedge$}{\string^}11 + x\texorpdfstring{$^\wedge$}{\string^}9 + x\texorpdfstring{$^\wedge$}{\string^}8 + x\texorpdfstring{$^\wedge$}{\string^}7 + x\texorpdfstring{$^\wedge$}{\string^}7 + x\texorpdfstring{$^\wedge$}{\string^}5 + x\texorpdfstring{$^\wedge$}{\string^}3 + x\texorpdfstring{$^\wedge$}{\string^}2 + x + x\texorpdfstring{$^\wedge$}{\string^}6 + x\texorpdfstring{$^\wedge$}{\string^}4 + x\texorpdfstring{$^\wedge$}{\string^}2 + x + 1 = x\texorpdfstring{$^\wedge$}{\string^}13 + x\texorpdfstring{$^\wedge$}{\string^}11 + x\texorpdfstring{$^\wedge$}{\string^}9 + x\texorpdfstring{$^\wedge$}{\string^}8 + x\texorpdfstring{$^\wedge$}{\string^}6 + x\texorpdfstring{$^\wedge$}{\string^}5 + x\texorpdfstring{$^\wedge$}{\string^}4 + x\texorpdfstring{$^\wedge$}{\string^}3 + 1 = y y modulo (x\texorpdfstring{$^\wedge$}{\string^}8 + x\texorpdfstring{$^\wedge$}{\string^}4 + x\texorpdfstring{$^\wedge$}{\string^}3 + x + 1) = x\texorpdfstring{$^\wedge$}{\string^}7 + x\texorpdfstring{$^\wedge$}{\string^}6 + 1.

The modular reduction by \doxylink{agent-base_2node__modules_2ms_2index_8js_a9e77e016b2928d7dcb493b89a0c9dc32}{m(x)} guarantees the result will be a binary polynomial of less than degree 8, so that it can fit in a byte.

The operation to multiply a binary polynomial b with x (the polynomial x in binary representation is 00000010) is\+:

b\+\_\+7x\texorpdfstring{$^\wedge$}{\string^}8 + b\+\_\+6x\texorpdfstring{$^\wedge$}{\string^}7 + b\+\_\+5x\texorpdfstring{$^\wedge$}{\string^}6 + b\+\_\+4x\texorpdfstring{$^\wedge$}{\string^}5 + b\+\_\+3x\texorpdfstring{$^\wedge$}{\string^}4 + b\+\_\+2x\texorpdfstring{$^\wedge$}{\string^}3 + b\+\_\+1x\texorpdfstring{$^\wedge$}{\string^}2 + b\+\_\+0x\texorpdfstring{$^\wedge$}{\string^}1

To get GF(b, x) we must reduce that by \doxylink{agent-base_2node__modules_2ms_2index_8js_a9e77e016b2928d7dcb493b89a0c9dc32}{m(x)}. If b\+\_\+7 is 0 (that is the most significant bit is 0 in b) then the result is already reduced. If it is 1, then we can reduce it by subtracting \doxylink{agent-base_2node__modules_2ms_2index_8js_a9e77e016b2928d7dcb493b89a0c9dc32}{m(x)} via an XOR.

It follows that multiplication by x (00000010 or 0x02) can be implemented by performing a left shift followed by a conditional bitwise XOR with 0x1b. This operation on bytes is denoted by xtime(). Multiplication by higher powers of x can be implemented by repeated application of xtime().

By adding intermediate results, multiplication by any constant can be implemented. For instance\+:

GF(0x57, 0x13) = 0xfe because\+:

xtime(b) = (b \& 128) ? (b \texorpdfstring{$<$}{<}\texorpdfstring{$<$}{<} 1 \texorpdfstring{$^\wedge$}{\string^} 0x11b) \+: (b \texorpdfstring{$<$}{<}\texorpdfstring{$<$}{<} 1)

Note\+: We XOR with 0x11b instead of 0x1b because in javascript our datatype for b can be larger than 1 byte, so a left shift will not automatically eliminate bits that overflow a byte ... by XOR\textquotesingle{}ing the overflow bit with 1 (the extra one from 0x11b) we zero it out.

GF(0x57, 0x02) = xtime(0x57) = 0xae GF(0x57, 0x04) = xtime(0xae) = 0x47 GF(0x57, 0x08) = xtime(0x47) = 0x8e GF(0x57, 0x10) = xtime(0x8e) = 0x07

GF(0x57, 0x13) = GF(0x57, (0x01 \texorpdfstring{$^\wedge$}{\string^} 0x02 \texorpdfstring{$^\wedge$}{\string^} 0x10))
